- ## [If LLMs are so good at coding… : r/LocalLLaMA _202606](https://www.reddit.com/r/LocalLLaMA/comments/1uf0oso/if_llms_are_so_good_at_coding/)
  - How come things like ROCm and the intel stack aren’t able to rapidly improve their software ecosystems to be a match for CUDA? Until the software from other vendors catches up with NVIDIA, they’re always going to get away with charging a massive premium on their “it just works” products.
- In my experience working with large, complex code bases, like the Linux kernel or Nvidia open source driver, LLMs aren’t great for writing the coding. However, recent agentic AI developments have made them helpful for understanding how the code works.

- AMD taking down ZLUDA was one of the most self-defeating moves in recent memory. One guy built the compatibility layer they needed and they killed it.
  - It wouldn't have been legal to ship it in the US. Thankfully, the guy is in Europe where there are no software patents and there is a legal right to reverse engineer for the purposes of interoperability (to protect against vendor lock-in). You know, consumer rights protection, the thing US politicians like to call over-regulation.

- Some people might remember this; https://www.phoronix.com/news/AMD-ZLUDA-CUDA-Taken-Down One man (Andrzej Janik) was able to develop CUDA compatible layer for AMD GPUs. ROCm wasn't even in their Top 10 priority list until last year.

- You’re assuming CUDA is standing still
- Tldr: Consistency. CUDA has run on basically every GeForce card for over a decade. Intel and AMD change or break the runtime all the time. That needs to stop.

- Because CUDA’s moat isn’t just code. It’s docs, tools, libraries, examples, bug history, StackOverflow answers, and a million cursed edge cases already solved.

- Because 90% of the coding data online is high-level end-user stuff. There genuinely isn't enough Kernel level coding data, whether it be common pitfalls, examples etc for AI to get trained on
